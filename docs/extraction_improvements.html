1. Executive Summary

Printo reads construction drawings and extracts ~26 structured fields (title block, room schedule, dimensions, materials), validates them against 18 rules, and pushes them to RealSoft ERP. This document explains how we improve extraction accuracy and which improvements are now integrated into the pipeline.

Two architectural decisions shape this phase: (1) all AI inference is routed through a pluggable AI sidecar (the Pratyaya / Ampris AI-aaS pattern) rather than calling a model API directly from Printo; and (2) every improvement is made measurable by a field-level benchmark harness so gains are provable, not assumed.

2. Current Pipeline & Gaps Addressed

The original pipeline sent the whole file to a vision model with a free-text JSON prompt and trusted the model's self-reported confidence. Three concrete gaps were found and fixed:

GapImpactFix
Deterministic prepass disabled (pdfplumber not installed) Free, reliable title-block text was never used Dependency added; prepass active + expanded
quantities validated (R10/R18) but never extracted Rules always warned; field always blank Added to schema, prompt, and prepass regex
Raw self-reported confidence Unreliable; confidence rules (R12–R15) noisy Replaced with a calibrated score (Section 5)

5. Confidence Calibration

Self-reported confidence is unreliable, so Printo computes a calibrated per-field score by combining independent signals:

  • Prepass agreement — a deterministic text-layer/OCR match sets the field to near-certain (0.97).
  • Format validity — drawing-number pattern, revision pattern, 1:NN scale: small boost if valid, penalty if not.
  • Cross-field consistency — e.g. sheet_number ≤ total_sheets; violations lower both scores.
  • Coverage floor — every populated field gets a sensible minimum so the UI always shows a score.

6. Sidecar Architecture

Printo never embeds a model key. ai_provider.py routes inference to a local sidecar (SIDECAR_URL, default http://localhost:8787) and falls back to built-in mock data when the sidecar is unreachable (AI_PROVIDER=auto). It supports a vision mode (sends the rendered image) and a text mode (sends OCR text), auto-selected from the sidecar's advertised capability.

EndpointContract
GET /health{ "status":"ok", "vision": true|false, "model": "..." }
POST /v1/extractbody { prompt, schema, image?{media_type,data(b64)}, text? } → { "data": { fields..., confidence } }
Swap-in ready: point SIDECAR_URL at the real Pratyaya AI-aaS (or any service implementing this contract) for genuine vision extraction — no Printo code change required. tools/mock_sidecar.py documents and stubs the contract.

7. Benchmark Methodology & Roadmap

benchmark.py runs the pipeline over the sample drawings and scores each field against ground_truth.json (bools by equality, lists by overlap, strings normalised with substring tolerance), printing per-file and overall accuracy. Run it before and after a change, or across providers, to quantify impact. Example: the mock provider scores 100% on the filename-aware samples; a fixed stub sidecar scores ~81%, correctly flagging the floor mismatches — demonstrating the harness detects real differences.

Tier-3 roadmap (next phase): golden-template matching, drawing-type routing, self-consistency voting, cross-sheet aggregation, and a human-in-the-loop feedback loop fed by the existing corrections table. Real accuracy figures require pointing the sidecar at a live vision model.
